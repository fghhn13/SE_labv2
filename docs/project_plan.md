## 项目计划（Minimal Emergent Structure Lab）

本文件描述本仓库的整体目标、v0 范围、关键里程碑、风险与验收标准。  
所有“实现状态”以实际代码为准，此处为规划。

---

### 一、项目目标

- **研究目标**：
  - 在 2D 栅格最短路径任务中，系统性探索：
    - 简单、可复用的决策结构（例如宏动作、子目标图、决策片段）是否会在学习过程中涌现；
    - 如何以人类可读的形式对这些结构进行提取与展示。
- **工程目标**：
  - 搭建一个**干净、模块化、可扩展**的实验平台：
    - 环境、Agent、结构服务、Trainer 相互解耦；
    - 日志与可视化通过回调挂载；
    - 方便后续扩展到更复杂任务和结构假设。
- **风格目标**：
  - 优先保证代码清晰与稳定接口，而非追求一开始就最强性能或最复杂模型。

---

### 二、v0 范围定义

v0 目标是得到一个**最小可运行且能收集基础结构信息**的版本，具体包括：

- **环境（envs）**
  - 一个基础的 2D 栅格最短路径环境：
    - 固定或简单可配置的地图尺寸。
    - 起点 / 终点位置。
    - 基本障碍布局（可固定或简单随机）。
  - 提供 `reset()` 和 `step(action)` 等基础接口。

- **Agent（agents）**
  - 至少一种基础 Agent 实现，例如：
    - 随机动作 Agent，或
    - 简单贪心 / 启发式 Agent。
  - 使用统一 Agent 接口，方便后续替换为更复杂模型。

- **Trainer（trainer）**
  - 一个能：
    - 运行多 episode；
    - 在每个 step 调用 Agent 和 Env；
    - 将轨迹信息发送给 `StructureService`；
    - 调用日志回调的最小 Trainer。

- **StructureService（structure）**
  - 一个“占位 + 最小功能”版本：
    - 能记录每条 episode 的路径（状态 / 动作 / 奖励）。
    - 能输出基础统计，例如成功 / 失败路径集合、平均路径长度等。
  - 为后续更复杂的结构抽象预留接口（例如 `update(trajectory)`、`export_structures()`）。

- **日志与可视化（logging / viz）**
  - v0 要求：
    - 有基础日志输出（终端或简单文件），记录关键指标。
    - 可选：简单的文本或图像化路径输出（例如打印栅格和路径）。
  - 不强求复杂前端或交互式工具。

- **CLI（cli）**
  - 至少一个入口脚本，用于：
    - 加载默认配置；
    - 运行指定 episodes 数的训练 / 采样；
    - 在结束时输出若干基础统计结果。

### 三、里程碑

#### 里程碑 1：架构骨架

- **目标**：建立清晰的目录结构与模块边界。
- **主要工作**：
  - 创建基础目录：`core/`、`envs/`、`agents/`、`structure/`、`trainer/`、`logging/`、`viz/`、`cli/`。
  - 定义关键接口 / 抽象：
    - `Agent` 接口（如 `act(observation, state)`）。
    - `Environment` 接口（如 `reset()`、`step(action)`）。
    - `StructureService` 接口（如 `on_step(...)`、`on_episode_end(...)`）。
    - 回调 / 事件接口。
  - 确定基础配置方式（例如简单的配置对象或 YAML / JSON 配置）。

#### 里程碑 2：首个可运行版本

- **目标**：实现最小可运行实验，从 CLI 启动、能在终端看到训练或采样过程的数据。
- **主要工作**：
  - 实现一个简单的栅格环境：
    - 支持 agent 在上下左右动作中移动。
    - 碰壁 / 障碍处理逻辑。
    - 终止条件与奖励设计（到达终点、步数上限等）。
  - 实现一个简单 Agent（例如随机走或基于曼哈顿距离的启发式）。
  - 实现最小 Trainer：
    - 能循环运行 N 个 episode；
    - 在每个 step 驱动 Env 与 Agent；
    - 在 episode 结束时输出基础统计（平均回报 / 成功率等）。
  - 实现一个 CLI 命令（或模块入口）：
    - 例如：`python -m cli.run_simple_grid_experiment`（名称可调整）。

#### 里程碑 3：结构观察版本

- **目标**：能够记录并导出基础“结构性数据”，支持初步观察。
- **主要工作**：
  - 完善 `StructureService`：
    - 支持记录每次 episode 的路径（状态 / 动作）和结果（成功 / 失败）。
    - 提供简单的结构摘要接口：
      - 常见路径集合；
      - 典型成功路径；
      - 平均路径长度等。
  - 与日志 / 可视化集成：
    - 将结构摘要打印到日志或另存为文件。
    - 可选：用简单可视化方式显示路径（文本栅格图或静态图像）。
  - 在 `quickstart` 文档中给出“如何查看结构输出”的示例。

#### 里程碑 4：实验就绪版本

- **目标**：可以较为方便地定义、运行和对比多个实验设置。
- **主要工作**：
  - 支持多种 Agent / 配置切换：
    - 例如基线 Agent vs 结构感知 Agent。
  - 提供实验配置机制：
    - CLI 参数或配置文件指定环境、Agent、训练步数、种子等。
  - 提供结果导出机制：
    - 保存多次 run 的统计结果与结构数据，便于后续分析。
  - 优化日志与可视化：
    - 提供基础指标曲线或对比表格。

### 四、风险与潜在坑

- **范围膨胀（Scope Creep）**
  - 风险：过早加入过多环境变种 / 复杂算法，导致核心基础设施迟迟不稳定。
  - 缓解策略：
    - 严格遵守 v0 范围，只实现最小可运行版本。
    - 新想法先写在文档 / issue 中，之后统一规划。

- **模块耦合过高**
  - 风险：Environment 与 Agent / StructureService 之间相互调用太多细节，导致后续难以替换或扩展。
  - 缓解策略：
    - 通过接口与抽象类约束依赖方向。
    - 优先保持 Environment 只负责状态转移与奖励。

- **结构表示过早复杂化**
  - 风险：一开始就追求复杂结构（如大规模图算法、复杂可视化），反而妨碍基础实验搭建。
  - 缓解策略：
    - 先从“记录路径 + 简单统计”开始；
    - 再逐步引入宏、子图等概念。

- **调试困难**
  - 风险：缺少清晰的日志与可视化，使得难以理解 Agent 行为和结构演化。
  - 缓解策略：
    - 从一开始就加入最简单的日志输出；
    - 在每个里程碑都检查“可观察性”是否足够。

- **性能过度优化**
  - 风险：在实验规模还很小时，就过早花费时间在性能优化上。
  - 缓解策略：
    - 优先保证代码清晰与结构合理；
    - 只有在实验需求明确受性能限制时再做优化。

### 五、验收标准

#### 里程碑 1：架构骨架

- 目录结构存在：`core/`、`envs/`、`agents/`、`structure/`、`trainer/`、`logging/`、`viz/`、`cli/`。
- 至少有一份接口定义文件（例如 Agent / Environment / StructureService 的抽象类或协议）。
- 文档中描述的层次关系与实际代码大体一致。

#### 里程碑 2：首个可运行版本

- 能在本地通过命令行运行一个脚本：
  - 命令执行后，不报错地完成至少 N 个 episode。
  - 终端输出基础统计（例如平均回报、成功率或路径长度）。
- 代码结构仍与架构设计基本一致（没有“临时写法”侵入核心模块）。

#### 里程碑 3：结构观察版本

- 运行实验后，能够：
  - 得到每条 episode 的路径记录（以某种可读格式保存或打印）。
  - 得到至少 1 种结构性摘要（例如最短成功路径、成功路径集合）。
- 文档中有说明“如何查看结构数据”的步骤。

#### 里程碑 4：实验就绪版本

- 能够通过修改配置或命令行参数：
  - 切换不同 Agent / 环境配置；
  - 重复运行多次实验并导出结果。
- 具备基础的对比能力：
  - 至少能够对比不同实验配置下的一两个核心指标（如成功率 / 平均路径长度）。
- 文档中给出一个“从零到完成一次对比实验”的完整流程。

---

### 六、后续扩展（超出 v0 范围，仅做占位）

以下内容不属于 v0 范围，仅作为未来可能方向的备忘：

- 更复杂的结构表示：
  - 基于图的宏结构挖掘；
  - 子目标自动发现；
  - 可解释策略抽取。
- 更复杂的环境：
  - 动态地图、随机障碍变化；
  - 多目标、多任务设置。
- 更复杂的算法：
  - 深度强化学习；
  - 基于规划 + 学习的混合方法。

这些部分在进入实现期前应当重新梳理需求与架构影响。
